# ROADMAP

План следующих улучшений проекта `Виджетрон`.

Документ фиксирует ближайшие инженерные и продуктовые шаги без попытки превратить его в полный backlog. Viewer не является активным приоритетом этого roadmap, если задача явно не требует обратного.

## Now

### 1. Декомпозиция `src/content.js`

- Цель: разрезать перегруженный `content.js` на несколько модулей с ясными границами ответственности.
- Зачем: сейчас в одном файле смешаны UX, DOM-capture, матчинг API, export/schema и storage-flow, что замедляет любые изменения и увеличивает риск регрессий.
- Результат: код выбора элемента, анализа DOM, ранжирования API, in-page UI и export-логики живёт в отдельных модулях с минимальными зависимостями между собой.
- Минимальный scope:
  - выделить `selection-ui`,
  - выделить `capture`,
  - выделить `matching`,
  - выделить `schema/export`,
  - сохранить текущее runtime-поведение.

### 2. Единый источник `OpenAPI schema` / export-логики

- Цель: убрать дублирование построения schema/export между `content.js` и `viewer.js`.
- Зачем: два независимых пайплайна почти гарантируют расхождение поведения normal-flow и Viewer.
- Результат: проект использует один канонический модуль для `OpenAPI schema`, sanitization examples и связанного export.
- Минимальный scope:
  - вынести общие helper-функции,
  - подключить их и из content-flow, и из Viewer,
  - не менять wire-format `captureBundle`.

### 3. Сценарные тесты для normal/pro flow

- Цель: закрыть тестами ключевые пользовательские сценарии, а не только helper-функции.
- Зачем: самые важные регрессии сейчас вероятнее всего лежат в склейке popup/content/background, а не в отдельных утилитах.
- Результат: есть автоматические проверки для основного happy-path и неоднозначных кейсов.
- Минимальный scope:
  - normal auto-select,
  - normal ambiguous manual choice,
  - pro manual flow,
  - smoke-проверка сохранения capture без изменения schema.

## Next

### 4. Release hygiene и единый `check` pipeline

- Цель: привести проект к более предсказуемому состоянию перед релизами.
- Зачем: уже видно расхождение между `manifest.json` и `package.json` по версии, а также не хватает единой команды проверки.
- Результат: понятная release-дисциплина и единый набор обязательных проверок.
- Минимальный scope:
  - выровнять версии `manifest.json` и `package.json`,
  - добавить `lint`,
  - добавить `npm run check`,
  - задокументировать это в README.

### 5. Дальнейшая доработка UX normal/pro без изменений Viewer

- Цель: сделать popup и in-page flow ещё понятнее и тише по UX.
- Зачем: normal-mode должен читаться как “включил → выбрал → готово”, а PRO должен оставаться расширенным, но не перегруженным.
- Результат: меньше вторичных сигналов, яснее статусы, плотнее layout, меньше инженерного жаргона.
- Минимальный scope:
  - продолжить упрощение popup copy,
  - доработать состояния normal/pro,
  - сократить шум в in-page подсказках и ambiguous-модалке,
  - не трогать Viewer.

### 6. Объяснимость выбора API

- Цель: показывать пользователю не только score, но и краткую причину выбора API.
- Зачем: это повышает доверие к normal-mode и помогает в ambiguous-кейсах.
- Результат: рядом с выбором API есть короткие пользовательские объяснения вроде “совпали данные ответа и текст блока”.
- Минимальный scope:
  - определить 3-5 простых причин выбора,
  - отобразить их без инженерной терминологии,
  - не превращать UI в диагностическую панель.

## Later

### 7. Фикстуры реальных capture payloads

- Цель: завести набор реальных сохранённых кейсов для регрессий.
- Зачем: синтетических тестов недостаточно для сложных JSON, truncation и multi-source widget cases.
- Результат: проект имеет стабильный fixture-набор для повторяемых проверок capture и schema-export.
- Минимальный scope:
  - простой single-source кейс,
  - ambiguous кейс,
  - multi-source кейс,
  - truncated/fallback кейс.

### 8. История последних захватов

- Цель: дать пользователю короткий локальный список недавних capture-результатов.
- Зачем: текущий `latestCapture` хорошо подходит для одноразового потока, но ограничивает повторное использование результатов.
- Результат: несколько последних capture доступны локально и управляются через TTL и явную очистку.
- Минимальный scope:
  - ограниченный локальный список,
  - TTL и ручная очистка,
  - без фоновой бесконечной истории,
  - без отправки данных наружу.
