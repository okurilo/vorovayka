# CODEX

## Назначение

- `CODEX.MD` — основной источник локальных правил и рабочих соглашений для Codex в этом проекте.
- История изменений и уже принятые решения фиксируются в `CHANGELOG.md`.
- Перед планированием, изменением кода и запуском команд нужно читать `CODEX.MD` и `CHANGELOG.md`.

## Базовые ограничения проекта

- Расширение остаётся продуктом уровня MVP и не должно превращаться в production-grade сетевой инспектор.
- Захват сети разрешён только после явного вооружения текущего `http/https` домена пользователем.
- Пользователь обязан иметь явный UI для включения и выключения armed-состояния без ручной правки storage.
- После вооружения домена захват должен переживать перезагрузку страницы, чтобы видеть ранние запросы.
- Глобальный фоновый захват на всех сайтах по умолчанию запрещён.
- Данные не отправляются наружу: разрешены только локальные механизмы расширения.
- Запись процесса допустима только поверх уже armed-домена, должна оставаться локальной и bounded по числу API-событий.

## Инварианты UX и данных

- `latestCapture` считается одноразовым сообщением и должен удаляться после чтения или по TTL.
- Пользователь должен иметь локальную кнопку копирования `latestCapture` без обязательного открытия viewer.
- Дефолтный экспорт во viewer должен оставаться компактным и пригодным для LLM: предпочтение у `DOM clean` и структурной схемы API, а не у полного raw payload.
- Если viewer или popup работают через compact fallback вместо полного capture, это должно быть явно показано в интерфейсе.
- DOM preview во viewer должен иметь стабильный fallback на `cleanHtml`; sandboxed стилизованный preview допускается только как дополнительный best-effort слой.
- Если для capture доступен `cloneSpec`, viewer должен показывать найденные связи `DOM ↔ API` с `JSON path` и источником endpoint.
- Страница записи процесса должна показывать пользователю, что именно накоплено: имя процесса, статус, число API-событий и экспортируемую хронологию.
- Публичный бренд продукта — `Виджетрон`; светлая бело-синяя визуальная система считается текущим направлением интерфейса.

## Проверка изменений

- После любых изменений кода обязательно запускать `npm test`.
- После изменений нужно проверять, что расширение не делает собственных внешних HTTP-запросов.
- Для security-правок приоритет у совместимости страницы: capture не должен ломать `fetch`/`XHR`.

## Обязательная фиксация изменений

- Каждая новая фича должна обновлять `README.md`, `CHANGELOG.md` и при необходимости `CODEX.MD`.
- Минимум нужно обновлять `CHANGELOG.md`, если меняются поведение, UI, ограничения или рабочие соглашения.
- Этот шаг обязателен даже для небольших правок.
